4. Merge Listed and Clinical Pools

将已分别完成清洗、疾病 Mapping 和库内收敛的“中国已上市池”与“中国临床池”，对齐到本次运行冻结的统一字段体系，完成跨库重复判断，生成权威 Full 基线，并从 Full 派生 Working Clean 或 Final Clean。

核心价值

两个来源库的字段、记录粒度和业务含义不同，不能简单上下拼接。

本 Skill负责：

检查两个来源库是否具备合并条件；

冻结本次运行的完整版和 Clean 版表头；

对齐上市侧和临床侧字段；

生成高召回的跨库候选；

关闭每条临床来源记录的保留、覆盖或排除决定；

建立唯一权威的 Full 基线；

支持临床来源“已上市”信息的异步补充；

从最新 Full 派生 Working Clean 或 Final Clean；

协调后续 BI 自有、MOA、VBP、TA 领先企业等模块结果的承接。

核心数据架构

Full

最终分子池_完整版 是唯一事实基线。

它应包含：

本次批准的完整宽表字段；

最终稳定编号；

上市与临床来源信息；

来源追溯；

后续补充信息；

后续子模块结果。

所有后续更新必须先进入 Full。

Working Clean

Working Clean 是当前 Full 的可用简表投影。

它可以在以下事项尚未关闭时生成：

临床来源已上市信息补充；

BI 自有产品判断；

MOA 重复判断；

VBP；

TA 领先企业等子模块。

但必须明确标注为 Working，而不能称为最终版。

Final Clean

Final Clean 是绑定到最新 Full 指纹的闭环简表，必须与 Full：

ID 集合一致；

行数一致；

顺序一致；

字段映射已批准；

必需补充事项已关闭；

必需子模块已完成、明确不适用或经批准跳过。

重要原则：列数不是固定常量

历史项目中可能出现 37 列、40 列完整版或 21 列 Clean 版，但这些只是已完成 TA 的运行实例。

每个 TA 都必须重新冻结：

wide_schema：完整版字段和两侧来源映射；

wide_fields：匹配、决策、来源和阶段所需逻辑字段；

clean_schema：Clean 核心字段及顺序；

clean_auxiliary_schema：可选的人工辅助字段。

不得把历史列数、列字母、行数或字段位置写成跨 TA 固定规则。

使用前提

上市侧应已完成：

适应症拆分；

TA Disease Mapping；

Mapping 人工复核；

同库 Mapping 行合并。

临床侧应已完成：

基础清洗；

临床资产和登记号整理；

Disease 标准化；

必要的库内收敛。

两个来源必须使用同一版本的 TA Disease 清单，并具备稳定来源 ID；若没有稳定 ID，必须明确批准一个绑定输入指纹的临时替代方案。

建议业务粒度

通常为：

资产或固定成分组合 × 剂型 × 映射 Disease

实际身份字段仍需按本次项目确认。盐型、立体异构体、释放形式和固定复方等差异不能被名称相似度自动抹平。

输入

需要提供：

当前 TA；

中国已上市输入文件及 Sheet；

中国临床输入文件及 Sheet；

两侧稳定来源 ID；

资产名称、通用名或主试验药；

剂型、Disease、适应症、阶段和登记号字段；

本次研究范围及时间边界；

Full 字段要求；

Clean 字段要求；

缺失值展示规则；

临床来源已上市信息补充规则；

必需的下游模块及其关闭状态；

新输出路径。

若临床侧有 Normalized Disease，可以按本次字段映射将其用于 相关适应症，避免上市侧是词组、临床侧是长句；但该字段名称和映射仍需逐次确认。

跨库候选思路

候选生成采用高召回策略，参考：

临床资产与上市药品名或通用名一致；

保守处理剂型后基础名称一致；

成分集合重合；

名称具有足够相似性；

没有直接候选。

一个临床来源行可能对应多个上市候选，因此：

候选行数 ≠ 临床来源行数

候选只是复核材料，不代表重复，也不触发自动删除。

跨库决策逻辑

判断顺序：

是否在研究范围
→ 是否为治疗性资产
→ 是否为独立资产或固定复方
→ 资产或成分是否一致
→ 剂型是否一致
→ Disease 是否一致
→ 适应症是否代表真实新增 Disease
→ 是否存在实质化学或释放形式差异
→ 上市库是否存在覆盖缺口

常见场景

场景

默认处理

同资产、同剂型、同 Disease

若适应症等价或为人群表达，临床行由上市行承接

同资产、同剂型、不同 Disease

先排查 Mapping 差异；真实新增 Disease 则保留

同资产、同 Disease、不同剂型

真实剂型差异通常保留

成分部分重合但含新研究分子

保留完整临床资产

仅将多个已上市药物作为非固定联合方案

排除该临床方案

固定复方

作为独立资产保留

诊断、检测或影像用途

作为非治疗性资产排除

临床库显示已上市但上市池无候选

进入上市信息缺口复核，不自动删除

登记号、公司、试验题目、目标顺序或“项目阶段=已上市”都不能单独决定保留或删除。

固定动作代码

每条唯一临床来源记录必须关闭为一个动作：

动作

含义

KEEP

保留临床行

DELETE_LISTED_COVERED

已上市池已覆盖同资产、剂型和 Disease

EXCLUDE_NON_THERAPEUTIC

非治疗性资产

EXCLUDE_NON_FIXED_REGIMEN

非固定联合方案

EXCLUDE_PRE_SCOPE_MARKETED

范围起点前已上市且无新增价值

HOLD

证据不足，不能进入正式 Full

正式生成 Full 前，HOLD、空白动作或未关闭状态必须为零。

标准流程

1. 只读预检

检查：

文件指纹、表头、有效范围和公式；

稳定来源 ID；

资产、剂型、Disease、适应症和阶段字段；

多值 Disease；

库内重复；

非治疗、非固定联合和上市信息缺口信号。

2. 冻结 Run Schema

逐字段确认：

Full 表头；

上市侧来源规则；

临床侧来源规则；

缺失值规则；

Clean 表头和顺序；

条件字段；

可选辅助列。

3. 生成统一待判断底稿

上市记录在前，临床记录在后；

保留数据来源；

为候选复核建立临时编号；

保存来源行 ID、原始行号和输入指纹；

不修改原始来源 Sheet。

4. 生成跨库候选

候选与决策文件绑定到一个不可变候选指纹。

5. 关闭跨库决策

每个唯一临床来源行只允许一个一致动作、修正值和 Rationale。一对多候选必须保持一致。

6. 预览并生成 Full

Full 行数应满足：

上市输入行
+ 临床输入行
- 被上市池覆盖的唯一临床行
- 被排除的唯一临床行

生成后：

最终编号一次性固定；

后续不再重排或重编号；

Full 成为唯一事实基线。

7. 并行推进后续工作

Full 锁定后可以并行：

Full
├─ Working Clean
├─ 临床来源已上市信息补充
└─ BI自有 / MOA / VBP / TA领先企业等子模块

不要求“临床已上市信息补充”必须在 Working Clean 前完成。

8. 回填晚到信息

补充信息返回后：

只按 Full 稳定 ID 匹配；

先更新 Full；

空白返回值不覆盖；

已有有效值冲突时必须取得覆盖批准；

更新补充状态；

将已有 Clean 标为过期；

从最新 Full 重新生成 Clean；

只复核真正受变更字段影响的子模块。

9. 生成 Final Clean

在补充信息和必需子模块关闭后，从最新 Full 派生最终简表。

临床来源已上市信息补充

常见候选条件为：

数据来源 = 中国临床库
且 项目中国阶段 = 已上市

实际触发规则必须在本次配置中确认。

建议状态：

待补充

已补充

未查到

不适用

经批准跳过

补充信息可能包括：

商品名；

持证商；

集团；

中国上市年代；

上市厂家数量；

其他经批准字段。

补充步骤不是 Working Clean 的硬前置，但若本次任务定义其为必需，则是 Final Clean 的关闭条件。

License Holder 条件逻辑

License Holder 不能简单固定为“上市取持证商、临床取申办方”。

推荐模式：

如果数据来源为中国已上市库
  → 取持证商
如果数据来源为中国临床库，项目已上市且已补到有效持证商
  → 取持证商
其他临床来源
  → 取申办方

如果已上市信息补充结果为 未查到，保留申办方，不虚构持证商。

子模块与辅助列

本 Skill 不计算：

BI 自有产品；

MOA 机制重复；

VBP；

TA 领先企业；

Local Giant；

商业优先级。

它可以：

在 Full 中承接已关闭的模块结果；

在 Working Clean 后追加运行特定的辅助列；

便于人工依据辅助列填写最终简表字段；

在模块关闭后从 Full 重新派生 Clean。

对简单模块，优先使用“聊天框返回命中 ID、人工按稳定 ID 回填”的轻量协作方式，不强制生成额外审计 Sheet。

输出

根据运行阶段可能包括：

统一合并待判断底稿；

跨库关系判断表；

中国已上市原始 Sheet；

中国临床原始 Sheet；

最终分子池_完整版；

Working Clean；

Final Clean；

临床已上市信息补充模板及隐藏审计状态；

隐藏运行清单和输入、候选、Full、Clean 指纹。

质量门槛与停止条件

必须停止的情况包括：

上游清洗或 Mapping 未完成；

两侧 Disease 词池版本不一致；

稳定来源 ID 无法建立；

Full 或 Clean 表头未批准；

库内重复未处理；

候选指纹变化；

一个临床来源行出现冲突动作；

存在 HOLD 或开放状态；

DELETE_LISTED_COVERED 没有有效上市承接行；

行数核算不成立；

Full 编号不唯一或不连续；

Clean 与 Full 的 ID、行数或顺序不一致；

Final Clean 仍存在必需补充或模块开放状态；

输出将覆盖输入或已有文件；

原始 Sheet、指纹或重新打开 QC 失败。

跨 TA 复用原则

可稳定复用：

Full 是唯一事实基线；

Clean 只能从 Full 派生；

高召回候选不等于删除；

每个临床来源行只能有一个关闭动作；

补充信息先回 Full；

Working Clean 与 Final Clean 分离；

子模块按稳定 ID 回填。

每个 TA 重新确认：

字段数量和表头；

来源字段映射；

业务粒度；

Disease 版本；

跨库候选和决策；

研究范围日期；

补充信息触发条件；

子模块要求；

输入、输出和最终行数。
